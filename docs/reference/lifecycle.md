# 生命周期模型

EvoPilot 的生命周期运行在持续演进控制面之上：

```text
Evidence -> Decision -> Approval -> Execution -> Validation -> Release Decision -> Learning
```

这个闭环不是一次性 agent 调用。运行证据先形成可追踪上下文，决策层识别机会和风险，治理层决定是否需要人工审批，执行层推进代码升级和 CI/CD，验证层产出独立证据，发布决策给出 `GO` / `CONDITIONAL-GO` / `NO-GO`，最后把结果沉淀为下一轮学习输入。

Dashboard Loop 执行页的 Source-to-GA 动态本体链路图把这条生命周期落到运行对象上：`SCM/Git Project` 对应源码和凭据边界，`Discovery Candidate` 对应证据输入，`Target Backlog` 对应目标和决策上下文，`Executor Graph` 对应执行编排，`Worker + Sandbox` 对应长任务连续性和受控执行边界，`Human Gate` 对应治理停点，`Source Closure` 对应源码写回，`CI/CD + Deploy` 对应交付和健康探测，`Release Decision` 对应产品发布判定，`GA Release` 对应最终可审计状态。它的作用是让用户在一个页面判断当前 loop 是缺项目/证据、缺目标、缺执行器、缺边界、缺审批、缺发布闭环，还是缺 GA 判定。

## 1. 项目接入

注册 AI Agent 产品、仓库提供方、运行证据通道、项目画像、环境、负责人和交付策略。

项目接入后会生成服务成熟度 Scorecard，评分维度包括：

- 项目注册验证。
- 证据覆盖。
- 治理闭环。
- 交付闭环。
- 自学习闭环。

Scorecard 用于判断项目是否已经具备进入生产进化流程的条件。

## 2. SLO 与治理策略

EvoPilot 会持续计算项目级 SLO 和治理策略：

- SLO 健康：基于延迟违规、失败发布和运行信号计算。
- 错误预算：判断是否还能继续自动进化或发布。
- 成熟度门禁：项目成熟度不足时只允许诊断或方案确认。
- 托管运行时供应链门禁：生产发布必须确认代码升级真实运行时的版本、镜像 Digest、SBOM、许可证报告、漏洞扫描报告和健康端点已经锁定。
- 成本预算门禁：基于 LLM / 工具调用成本、Token 数和高成本事件判断是否允许继续自动进化。
- 运行时供应链报告：逐项展示代码升级执行器等托管运行时是否已满足产品生产级锁定；GitHub Actions/GitLab CI 通过项目 DevOps preflight 做 provider、token、workflow/check/job 和健康探测校验。
- 发布就绪度：把用户确认、代码升级、CI/CD、SLO、成本和运行时供应链合成为放行、需确认或阻断结论。
- 灰度策略：根据发布就绪度、SLO 灰度窗口、成本灰度窗口和回滚准备判断是否允许 Canary。

当错误预算燃烧或策略失败时，系统会降低自动化等级，并要求人工确认。

## 3. 基线评估

建立当前产品基线：

- 任务成功率。
- 失败分类。
- 响应质量。
- 工具失败率。
- 延迟。
- Token 与模型成本。
- 用户满意度。
- 发布健康度。

## 4. 运行观测

采集会话、工具调用、trace、MCP 事件、CI 信号、部署信号、用户反馈和生产指标。

## 5. 证据打包

将原始信号标准化为带可追溯性和时间窗口的证据包。

## 6. 证据聚类与归因

系统先把同一条链路、同一模块或同一类运行信号聚合为证据簇，并计算动态基线。

- 性能类证据会对比 p95、最大耗时和用户定义阈值。
- 工具类证据会识别失败、恢复和降级路径。
- RAG、评测、成本、安全和用户反馈会作为交叉印证信号。
- 输出内容包括证据摘要、失败归因、置信度理由和命中策略。

## 7. 机会挖掘

识别产品缺口、性能热点、可靠性风险、测试缺口、文档漂移、策略失败和模块边界异味。

## 8. 自学习机会发现

系统会把运行证据自动沉淀为 Eval Dataset，并在后台生成机会洞察：

- 自动评估机会置信度、影响面、证据数量、关联评测集和发布后学习记录。
- 自动给出机会洞察分数，用于排序哪些问题应该优先进入进化方案。
- 自动区分人工导入评测集与系统智能沉淀评测集。
- 每次发布结果会反向影响后续机会洞察质量，成功进化会增强相似机会的可信度，失败进化会提示收紧验证契约。

## 9. 策略排序

应用项目专属权重、风险策略、保护路径和交付约束。

## 10. 治理门禁

系统会按项目策略决定自动化等级：

- 诊断模式：证据不足或影响面不明确，只生成诊断结果。
- 方案确认：形成进化方案，但必须由用户确认后才能执行。
- 人工设计：风险较高或影响受控资产，需要人工设计后推进。
- 自动执行：仅在低风险、高置信度、高优先级且项目策略允许时进入自动执行。

门禁评估结果会同时进入 `/api/v1/governance/policy-evaluations`、`/api/v1/release/readiness`、Prometheus 风格指标和 Dashboard 首页。生产模式下，运行时供应链锁定不完整、成本超预算、SLO 错误预算耗尽都会阻断自动发布。

## 11. 代码感知方案生成

机会点不会直接变成执行方案。EvoPilot 会先根据项目注册信息读取当前 Git 基线代码，形成代码上下文包：

- 仓库分支和提交信息。
- README、运行入口、依赖声明、服务入口、测试脚本和关键源码。
- 项目运行配置、验证命令和受保护路径。
- 关联评测集、证据指标和目标约束。

软件架构师能力基于这份上下文输出 Markdown 进化方案。方案必须包含当前代码事实、可行性判断、修改范围、验证计划、风险与回滚。生产模式下，如果项目未注册、仓库不可访问、代码基线无法读取或真实 LLM 不可用，方案生成必须失败，不能降级为模板方案。

## 12. 计划生成

生成包含预期效果、影响面映射、验证契约、回滚路径和评审报告的演进计划。

## 13. 评审与确认

产品、架构、风险和交付评审人决定是否继续推进。

## 14. 交付

创建 PR/MR，运行 CI，执行 CD，管理发布、灰度、回滚和发布证据。

## 15. 进化验证契约

每个进化方案都会生成验证契约，并按机会点类型选择验证集：

- 语义回归：确认核心任务结果没有退化。
- 性能回归：确认 p95、最大耗时或吞吐达到目标。
- 成本回归：确认 Token、模型或工具成本没有恶化。
- 安全回归：确认鉴权、密钥、受保护路径和高危发现为零。
- 冒烟测试与功能闭环测试：确认从证据到交付的产品链路仍可运行。

## 16. 发布后验证

发布前会先计算发布就绪度：

- `READY`：允许进入灰度、A/B 或正式发布。
- `NEEDS_APPROVAL`：仍可推进，但需要负责人确认灰度范围和回滚策略。
- `BLOCKED`：存在失败门禁，不能进入 CI/CD 后续发布阶段。

随后系统会计算灰度策略：

- `CANARY`：允许从 10% 流量开始灰度，持续观察 SLO、成本和用户反馈。
- `MANUAL_APPROVAL`：只允许 1% 小流量灰度，并要求负责人确认。
- `BLOCKED`：发布就绪度、SLO、成本或回滚准备不满足要求，不允许进入灰度。

验证预测效果是否真实发生。

## 17. 学习

持久化决策、结果、失败假设、采纳规则、拒绝规则和未来优先级调整。
